## Architecture Summary

### Four Pillars

1. **The Ear (Notifier)** - FastAPI webhook receiver at `src/notifier_service.py`
   - Receives GitHub webhooks at `/webhooks/github`
   - Validates HMAC signatures
   - Detects task intent (Plan/Bug/Feature)
   - Applies `agent:queued` label

2. **The State (Queue)** - GitHub Issues as database
   - Labels represent status: `agent:queued`, `agent:in-progress`, `agent:success`, `agent:error`
   - Assignees provide distributed locking

3. **The Brain (Sentinel)** - Background polling service at `src/orchestrator_sentinel.py`
   - Polls for queued items every 60 seconds
   - Claims tasks via assign-then-verify pattern
   - Dispatches workers via shell bridge

4. **The Hands (Worker)** - DevContainer-based AI execution
   - Executes via `scripts/devcontainer-opencode.sh`
   - Runs opencode CLI with AI agents

### Data Flow

```
GitHub Webhook → Notifier → Apply agent:queued label
                                    ↓
Sentinel polls → Claims task → Dispatches Worker
                                    ↓
Worker executes → Returns exit code → Sentinel updates status
```

## Key Patterns

### Assign-Then-Verify Pattern
Used for distributed locking when claiming tasks:
1. Assign sentinel bot to issue
2. Re-fetch issue
3. Verify assignment before proceeding

### Polling-First Resiliency
Webhooks are optimization; polling is primary:
- Enables automatic state reconciliation on restart
- Self-healing against network partitions

### Shell-Bridge Protocol
Orchestrator interacts with Worker exclusively via `devcontainer-opencode.sh`:
- Commands: `up`, `start`, `prompt`, `stop`
- Clear separation between Logic Layer and Infrastructure Layer

### Secret Scrubbing
Before posting logs to GitHub:
- Remove patterns: `ghp_*`, `ghs_*`, `sk-*`, `zhipu_*`, Bearer tokens
